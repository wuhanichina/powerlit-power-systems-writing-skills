# Letter Evidence and Conclusion Reference

Use this when drafting the case-study, numerical-results, or conclusion section of an IEEE power-system Letter.

## Corpus Signal

In the 69-paper Letter corpus:

- case/result sections appear in 95.7% of Letters;
- median case/result paragraphs: about 7;
- conclusion sections appear in 97.1%;
- median conclusion paragraphs: 1;
- common case headings include `III. CASE STUDY`, `IV. CASE STUDIES`, `IV. NUMERICAL RESULTS`, and `IV. SIMULATION RESULTS`;
- common conclusion headings are `IV. CONCLUSION` or `V. CONCLUSION`.

## Evidence Principle

Letter evidence should be minimal but decisive. It verifies the one hard claim; it does not imitate a full paper's experiment package.

Good evidence forms:

- analytical derivation plus one numerical illustration;
- counterexample plus implication;
- small benchmark plus one runtime/accuracy comparison;
- standard test system plus one focused figure or table;
- one sensitivity only when it is central to the claim.

## Case / Results Structure

Use this compact rhythm:

1. Minimal system/setup.
2. One result figure/table.
3. One interpretation tied to the claim.
4. Optional baseline or sensitivity.

Do not add multiple benchmark families, broad ablation, or long scenario sweeps unless they are the core contribution.

## Conclusion Structure

Use one paragraph by default:

1. State what this letter established.
2. Name the core deliverable: formula, counterexample, model, algorithm, or index.
3. State the key numerical or analytical support.
4. Add one boundary or future extension only if needed.

Avoid:

- repeating a full contribution list;
- broad future work;
- unsupported impact claims;
- vague "effectiveness and superiority" language.

## Sentence Patterns

Use patterns as logic, not boilerplate:

- `This letter established that ...`
- `The derived expression shows ...`
- `The counterexample demonstrates ...`
- `Numerical results on ... verify ...`
- `Future work will extend ... to ...` only if this boundary matters.
